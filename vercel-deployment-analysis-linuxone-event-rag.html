Vercel Deployment Analysis

Repository: github.com/IBMThinkBOB/LinuxONE-Event-RAG  ·  Analysed 15 Jul 2025

Frontend on Vercel
✓ Yes — as-is

Vite/React SPA, static build, no server required.

Backend on Vercel
✗ No — hard blockers

Requires local Ollama, pgvector, 500 MB+ ML models, and persistent state.

Recommended architecture: Deploy frontend to Vercel. Host the backend + Ollama on a long-running server (IBM LinuxONE, Fly.io, Railway, Render, or a VPS). The frontend's /api proxy just needs to point at the backend URL.

Why the backend cannot run on Vercel serverless

Blocker 1 — Ollama dependency: Every query calls http://localhost:11434/api/generate. Ollama is a separate process that must be running on the same host. Vercel Functions have no sidecar processes.
Blocker 2 — ML model bundle size: sentence-transformers/all-MiniLM-L6-v2 (~90 MB) and cross-encoder/ms-marco-MiniLM-L-6-v2 (+90 MB) are downloaded at startup. Vercel's function bundle limit is 50 MB (250 MB compressed). PyTorch alone exceeds this.
Blocker 3 — PostgreSQL + pgvector: The backend requires a running PostgreSQL 15+ instance with the vector extension enabled and an HNSW index. Vercel has no built-in Postgres with pgvector (Vercel Postgres is vanilla).
Blocker 4 — Execution timeout: Vercel serverless functions time out at 60 s (Hobby) / 300 s (Pro). A single RAG query involves embedding generation + reranking + LLM inference, which typically takes 15–120 s depending on Qwen model size.
Blocker 5 — No persistent in-memory state: EmbeddingService and RerankingService are cached as global singletons. Serverless functions are stateless and cold-start on every invocation, reloading the 90 MB models each time.

Conclusion: The backend requires a locally running Ollama server on the same host. It uses raw HTTP (requests.post) to localhost:11434 — not a managed API. There is no API key, no cloud endpoint, and no fallback. Ollama must be installed and the Qwen model pulled before the backend can answer any query.
Possible mitigation: Replace the Ollama call with an OpenAI-compatible or IBM watsonx API (e.g. ibm-watsonx-ai SDK). This would remove the local-Ollama dependency but requires a cloud LLM account and prompt/token costs.

4. RAG Infrastructure

QuestionAnswer
Vector database PostgreSQL 15 + pgvector extension (HNSW index, 384-dim cosine similarity). Defined in backend/scripts/init_db.sql.
Embeddings stored on disk? No. All embeddings live in the chunks.embedding vector(384) column in PostgreSQL. No flat files.
Does the app write files locally? Only evaluation/test scripts write JSON reports (evaluate_hybrid_rag.py, evaluate_retrieval.py). The main API does not write to disk.
Persistent storage required? Yes — PostgreSQL with pgvector must persist the documents and chunk embeddings across restarts. A stateless deployment with ephemeral storage would lose all indexed data.
Ingestion job limits Yes. Ingestion (backend/scripts/ingest_documents.py) runs PDF parsing + batch embedding generation. For 5 Redbooks the QUICKSTART estimates 5–10 minutes. Serverless functions cannot run for more than 5 minutes.
Model download at startup Yes. sentence-transformers/all-MiniLM-L6-v2 (~90 MB) downloads from HuggingFace Hub on first run. This is unsuitable for cold-start serverless.
No Dockerfile present? Correct — no Dockerfile in the repo. docker-compose.yml is present but only spins up PostgreSQL. The backend is run directly with uvicorn.

5. Deployment Assessment

Frontend on Vercel — ✓ Yes, with one env-var change

  • Vite builds a static SPA into dist/ — Vercel hosts this natively.
  • The /api proxy in vite.config.js is dev-server only and not included in the build.
  • Before building, set VITE_API_BASE_URL=https://your-backend.example.com and update frontend/src/services/api.js to use it.
  • Add a vercel.json (see template above) for SPA routing rewrites.

Backend on Vercel — ✗ No, 5 hard blockers

  • Requires a running Ollama process on the same machine (no cloud API).
  • Python ML packages (PyTorch, sentence-transformers, scikit-learn) far exceed the 50 MB bundle limit.
  • Needs a PostgreSQL + pgvector instance (not Vercel Postgres, which lacks the vector extension).
  • Query latency (embedding + reranking + LLM) can exceed 60 s timeout on Hobby plan.
  • Singleton ML model cache is incompatible with stateless cold-start invocations.

Recommended: Split Deployment

LayerHostNotes
Frontend (React/Vite) Vercel Static build. Free tier works. Point VITE_API_BASE_URL at backend.
Backend (FastAPI + Uvicorn) Fly.io / Render / Railway / VPS Needs 2–4 GB RAM for ML models. Always-on container required.
LLM (Ollama + Qwen) Same VM as backend, or GPU cloud Must be co-located with backend or reachable on private network. OLLAMA_BASE_URL env var controls the endpoint.
Database (PostgreSQL + pgvector) Supabase / Neon / self-hosted Both Supabase and Neon support the pgvector extension. Set DATABASE_URL env var.
If you want cloud LLM: Swap the Ollama HTTP call in backend/app/services/llm_service.py for an OpenAI-compatible endpoint (e.g. IBM watsonx, together.ai, Groq). The base_url and model are already parameterised via config.py — the change is minimal.
